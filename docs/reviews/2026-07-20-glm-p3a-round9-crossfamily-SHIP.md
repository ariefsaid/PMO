### Findings

**SHOULD-FIX 1 — Webhook inbound adopt has no in-flight guard (the sweep's B5 fix is not mirrored on the webhook path).**
`supabase/functions/erpnext-webhook/index.ts` (whole file) → `applyErpFeedEvent` → `applyInboundChange`. The sweep skips any ERP doc whose anchor carries an unresolved outbox key (`createInFlightAnchorProbe`); the webhook does not. Concrete path: dispatch POSTs a Sales Invoice (outbox `committing`); ERP commits and fires the webhook immediately; the webhook's `claimExternalRef` (fresh UUID ← ERP name) lands before the dispatch's fenced `record_outbox_ref` (R ← ERP name). The outbox insert then hard-fails 23505 on `external_refs_org_domain_extid_key` (0093) — a *different* constraint than its `on conflict (org_id,domain,pmo_record_id)` target, so it is not caught — stranding the row at `committed` and leaving a phantom mirror under a random UUID; the user's SI R is unlinked. **Why SHOULD-FIX, not BLOCK:** no money is lost (the phantom reflects the correct ERP amount), no money is duplicated (the `one_inflight_per_record` partial index refuses any retry create for R), and there is no cross-tenant leak (HMAC + company scoping still held). It is a real, occasionally-reachable race that degrades to an operator-recoverable stuck state. Fix mirror: route the webhook's adopt candidates through the same `createInFlightAnchorProbe` the sweep uses (or have `record_outbox_ref` fall back to `on conflict (org_id,domain,external_record_id) do update set pmo_record_id = …` so the outbox reclaims an existing webhook mapping).

**SHOULD-FIX 2 — Quarantined-past-window replay can REISSUE (immutable anchor) without re-asserting the actor's CURRENT authz.**
`supabase/functions/adapter-dispatch/authGuard.ts:169-170` — `replayMayIssueErpWrite` returns true only for `pending`/`failed`, so a `quarantined` candidate skips `checkOutboxReplayAuthorization`. But `dispatch.ts` `claimAndCommit(isRecoveryReissue=true)` will, for an immutable-anchor kind whose post-window probe misses, fall through to a fresh ERP POST (only mutable-anchor/PE rows are HELD). The authGuard comment claims quarantined rows only "adopt-or-hold"; that premise is false for immutable anchors. Concrete path: Admin A dispatches a Purchase Invoice; the POST is slow, row goes `committing` → stale → `quarantined`; within ~6 min A is demoted/deactivated; sweep claims the quarantined row past its window, probe misses (the original POST genuinely failed), immutable anchor → REISSUE → mints the PI under a no-longer-authorized actor. **Why SHOULD-FIX, not BLOCK:** the body is the ORIGINAL authorized payload (M-3 digest-bound), the amount/party/tenant are unchanged, and domain revocation IS still caught by `reconcileOrgOutbox`'s `owned.has(domain)` pre-filter — so this is an authz-freshness gap on a recovery reissue, not money loss / duplication / self-approval / cross-tenant. Fix: include the quarantined state in `replayMayIssueErpWrite` (the reissue is a new money write), or re-check authz inside `claimAndCommit` when `isRecoveryReissue && probed === null`.

**NIT — Master-data (Supplier/Customer) inbound adopt is deliberately unscoped by company.**
`companyScope.ts` — `isCompanyScopedKind` excludes `supplier`/`customer` (they are ERP-site-global masters), so on a hypothetical multi-PMO-tenant-shared ERPNext site both tenants would mirror the full party list. Not reachable today (single-tenant-per-ERP-site is the deployed config; no org employs ERPNext in prod) and leaks party *names* not money. Defense-in-depth for a future multi-tenant ERP site.

### What I checked and found clean

- **Outbox concurrency in the round:** partial unique index (one non-terminal row/record) + single-use key index + 4-tuple unique + claim budget (`MONEY_COMMIT_CLAIM_BUDGET_MS` 60s, asserted vs quarantine 300s) + probe budget (`ERP_PROBE_TIMEOUT_MS` 20s, `maxRetries 0`) + `commitDeadlineAtMs` POST refusal. Approver≠author is checked against the append-only `sales_invoice_authors` set, fail-closed on null author. `claim_sales_invoice_author` (callable by `authenticated`) can only append the caller — a self-DoS at worst, never a bypass.
- **Company scoping symmetry:** webhook per-event gate + sweep per-row + server-side filter all delegate to the one `companyScope.ts`; no-company binding fails closed (skip), doc-stating-no-company fails closed (refuse). Money docs only; masters intentionally unscoped (NIT above).
- **Recovery probe:** anchor `like` is `escapeLikePattern`-escaped; PE composite probe conjoins `payment_type` + `party_type` server-side and re-validates post-fetch; the fallback anchor probe carries the discriminator via `withPaymentTypeDiscriminator` in both the sync path (index.ts) and the sweep (buildOutboxProbe). A Receive recovery cannot adopt a Pay doc and vice-versa.
- **Key–payload binding:** `canonicalCommandDigest` is recomputed on the sweep from the persisted payload and compared via `payloadDigest`; the `created_after` window-bound is excluded. Key reuse with a mutated amount/party/refs is rejected, not silently reconciled.
- **Webhook trust boundary:** HMAC is sole trust; stream-bounded body cap; ambiguous multi-org signature match → reject; DB-load failure → retryable 500 (not a silent 401); domain ownership + company re-checked after HMAC.
- **Cross-org FK guards:** dispatch pre-flight (`assertCommandLinksSameOrg`) AND service-role mirror writers (`requireOwnOrgLink`/`resolveLinkOrNull`) cover all six link fields across revenue + procurement; a vanished-in-TOCTOU link nulls gracefully, a genuine cross-org link throws `cross-org-link-rejected`.
